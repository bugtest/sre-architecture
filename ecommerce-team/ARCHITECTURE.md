## 一、设计哲学

### Linear Pipeline vs Flywheel：本质差异

| 维度 | 线性Pipeline | 飞轮架构（电商团队） |
|------|---------|---------|
| 拓扑 | 线性Pipeline（Phase 1→N） | **闭环飞轮**（数据反馈驱动迭代） |
| 时间尺度 | 单次执行，产出文档 | **持续运营**，周期性决策 |
| 输入 | 固定输入 | 市场信号（动态） |
| 输出 | 文档（终态） | 运营动作（持续优化） |
| 核心逻辑 | 一次性任务 | 持续迭代优化 |

**升维结论：电商系统不能用Pipeline，必须用飞轮+事件驱动架构。**

## 三、七大角色定义

### Role 01: CONDUCTOR（指挥中枢）

**本质**：不是执行者，是**决策路由器 + 冲突仲裁者**

| 属性 | 值 |
|------|---|
| 触发条件 | 用户指令 / 定时任务 / 异常告警 |
| 核心能力 | 任务分解、依赖编排、资源分配、冲突仲裁 |
| 输出 | 调度指令（谁做什么、什么顺序、超时策略） |
| 决策权 | 角色间结论冲突时的最终裁决 |

**关键机制**：
- **依赖图引擎**：自动识别任务间依赖，最大化并行度
- **冲突仲裁协议**：当SCOUT说"该品类红海"但RADAR说"有增量空间"时，CONDUCTOR基于数据权重裁决
- **熔断机制**：单个Agent超时/失败时，降级执行而非阻塞全链路

---

### Role 02: RADAR（市场雷达）

**本质**：**市场信号采集与趋势预判**

| 属性 | 值 |
|------|---|
| 输入 | 品类关键词 / 竞品URL / 行业赛道 |
| 核心任务 | 趋势捕捉、需求缺口挖掘、竞品动态监控 |
| 输出 | `RADAR-REPORT.md`（市场信号图谱） |
| 数据源 | 电商平台数据、社交媒体趋势、搜索指数 |

**输出结构**：
```markdown
## 市场信号报告
### 1. 趋势信号 [置信度: HIGH/MEDIUM/LOW]
- 上升趋势: ...（数据源 + 时间窗口）
- 衰退信号: ...
### 2. 需求缺口
- 未被满足的用户痛点（来源: 差评分析/社交讨论）
### 3. 竞品异动
- 新品上架 / 价格调整 / 流量变化
### 4. 风险预警
- 政策风险 / 季节性衰退 / 供应链波动
```

**红队自检**：
- ❓ 趋势是真实需求还是平台算法推荐泡沫？
- ❓ 数据时间窗口是否足够长（≥3个月）以排除噪声？
- ❓ 是否遗漏了线下渠道的竞争信号？

---

### Role 03: SCOUT（选品侦察）

**本质**：**从市场信号中提炼可执行的选品决策**

| 属性 | 值 |
|------|---|
| 输入 | RADAR-REPORT.md + 供应链约束 + 利润要求 |
| 核心任务 | 品类评估、竞争度分析、供应链可行性验证 |
| 输出 | `SCOUT-PICKS.md`（选品清单 + 优先级排序） |
| 决策框架 | 市场容量 × 竞争度⁻¹ × 利润率 × 供应链难度⁻¹ |

**评估矩阵**：
```
每个候选品 → 4维评分（1-10）:
  M: Market Size（市场容量）
  C: Competition（竞争度，反向）
  P: Profit Margin（利润空间）
  S: Supply Chain（供应链可控性）

综合分 = M × (10-C) × P × S / 1000
阈值: ≥ 3.0 进入候选池
```

**红队自检**：
- ❓ 是否存在"数据看起来好但实际被头部垄断"的伪蓝海？
- ❓ 供应链验证是否基于实际询价而非估算？
- ❓ 季节性因素是否已纳入（避免选了一个Q4爆品但现在是Q1）？

---

### Role 04: FORGE（内容锻造）

**本质**：**将产品卖点转化为高转化内容资产**

| 属性 | 值 |
|------|---|
| 输入 | SCOUT-PICKS.md + 产品参数 + 竞品listing分析 |
| 核心任务 | 标题优化、卖点提炼、详情页文案、A+/图文内容 |
| 输出 | `FORGE-CONTENT.md`（全套内容资产） |
| 方法论 | FAB法则（Feature→Advantage→Benefit）+ 搜索词嵌入 |

**内容产出清单**：
```markdown
## 内容资产包
### 1. 标题矩阵
- 主标题（核心关键词 + 核心卖点，≤60字符）
- 副标题变体 ×3（不同关键词组合）
### 2. 五点卖点（Bullet Points）
- 每条: 痛点场景 → 产品方案 → 用户获得感
### 3. 详情页结构
- Hook（3秒抓住注意力）
- 痛点共鸣（用户语言，非技术语言）
- 解决方案展示
- 信任背书（数据/认证/对比）
- CTA（行动号召）
### 4. 搜索优化
- 核心关键词（搜索量 + 竞争度）
- 长尾词布局策略
- 后台Search Terms
```

**红队自检**：
- ❓ 卖点是"我们觉得好"还是"用户搜索词验证的需求"？
- ❓ 是否对标了Top 3竞品的内容策略并做出差异化？
- ❓ 文案是否通过了"删掉品牌名还能认出是谁"的独特性测试？

---

### Role 05: BLADE（定价利刃）

**本质**：**利润最大化的动态定价策略引擎**

| 属性 | 值 |
|------|---|
| 输入 | SCOUT-PICKS.md + 成本结构 + 竞品价格带 |
| 核心任务 | 成本拆解、价格带定位、促销策略、利润模拟 |
| 输出 | `BLADE-PRICING.md`（定价方案 + 利润模型） |
| 核心公式 | 目标售价 = 成本 × (1 + 目标利润率) / (1 - 平台扣点 - 广告费率) |

**定价框架**：
```markdown
## 定价决策
### 1. 成本瀑布图
- 采购成本 → 物流 → 包装 → 平台佣金 → 广告 → 退货准备金
### 2. 竞品价格带分析
- P10/P25/P50/P75/P90 价格分布
- 各价格带的评论量/评分分布（≈市场份额代理指标）
### 3. 定价策略
- 渗透定价 / 撇脂定价 / 锚定定价 — 选哪个 + 为什么
### 4. 促销矩阵
- 大促节点 × 折扣力度 × 利润底线
### 5. 盈亏模拟
- 场景A: 日销X件 × 售价Y → 月利润Z
- 场景B/C: 悲观/乐观模型
- Break-even点: 日销量阈值
```

**红队自检**：
- ❓ 成本是否包含了隐性成本（退货率、仓储滞销、广告ACOS恶化）？
- ❓ 竞品价格数据是实时的还是历史快照？
- ❓ 定价是否考虑了平台比价算法对流量的影响？

---

### Role 06: PULSE（数据脉搏）

**本质**：**运营数据采集、异常检测、归因分析**

| 属性 | 值 |
|------|---|
| 输入 | 运营数据 / 广告数据 / 用户行为数据 |
| 核心任务 | 数据看板、异常告警、归因分析、A/B测试评估 |
| 输出 | `PULSE-REPORT.md`（数据洞察 + 行动建议） |
| 核心指标 | CTR, CVR, ACOS, ROAS, LTV, 复购率 |

**分析框架**：
```markdown
## 数据脉搏报告
### 1. 核心指标仪表盘
- 流量指标: UV / PV / 跳出率
- 转化指标: 加购率 / 下单率 / 支付转化率
- 广告指标: ACOS / ROAS / CPC / 展示份额
- 利润指标: 毛利率 / 净利率 / 库存周转天数
### 2. 异常检测 [自动]
- 指标偏离基线 ≥2σ → 触发告警
- 归因路径: 指标异常 → 可能原因 × 3（按概率排序）
### 3. 竞对动态关联
- 自身指标变化 ↔ 竞品动作的时间线对照
### 4. 行动建议
- 每条建议: [优先级] + [预期影响] + [执行成本]
```

**红队自检**：
- ❓ 指标变化是自身原因还是平台大盘波动？
- ❓ 归因是相关性还是因果性？（A/B测试验证了吗？）
- ❓ 数据采集频率是否匹配决策周期？（日数据做周决策 vs 时数据做日决策）

---

### Role 07: ORACLE（决策先知）

**本质**：**汇总全链路信息，输出最终运营决策**

| 属性 | 值 |
|------|---|
| 输入 | 所有其他角色的输出报告 |
| 核心任务 | 综合研判、优先级排序、资源分配建议、风险预案 |
| 输出 | `ORACLE-DECISION.md`（执行决策书） |
| 决策权 | 最终建议权（非执行权，执行权在老板） |

**决策书结构**：
```markdown
## 执行决策书
### 0. 一句话结论
> 做什么 + 为什么 + 预期结果

### 1. 决策矩阵
| 选项 | 预期收益 | 风险等级 | 资源需求 | 推荐度 |
|------|---------|---------|---------|--------|

### 2. 执行路线图
- Week 1-2: ...
- Week 3-4: ...
- Month 2-3: ...

### 3. Kill Criteria（止损线）
- 如果指标X在Y天内未达到Z → 执行Plan B / 止损退出

### 4. 风险预案
- Risk 1 → Mitigation
- Risk 2 → Mitigation

### 5. 资源需求
- 资金 / 人力 / 工具 / 时间
```

**红队自检**：
- ❓ 决策是否过度依赖单一数据源？
- ❓ 是否存在"所有角色都说好但常识告诉你不对"的集体盲区？
- ❓ Kill Criteria是否足够具体可量化（避免"情况不好就停"的模糊标准）？

## 四、飞轮运行模式

### 模式A: 新品上架（Full Pipeline）

```
CONDUCTOR 接收指令: "评估并上架品类X"
    │
    ├── [并行] RADAR(市场调研) + SCOUT(竞品侦察)
    │         ↓ 输出合并
    │     CONDUCTOR 仲裁: 确认品类可行性
    │         ↓
    ├── [并行] FORGE(内容制作) + BLADE(定价策略)
    │         ↓ 输出合并
    │     CONDUCTOR 审核: 内容×定价一致性检查
    │         ↓
    └── ORACLE: 输出执行决策书 → 老板审批
              ↓ 上架后
          PULSE: 持续监控 → 异常反馈 → 触发迭代
```

### 模式B: 日常运营（Event-Driven）

```
PULSE 检测到异常（CVR下降15%）
    │
    CONDUCTOR 启动诊断流程:
    ├── RADAR: 检查市场/竞品是否有变化
    ├── FORGE: 检查内容是否需要更新
    ├── BLADE: 检查价格竞争力
    │         ↓
    ORACLE: 归因 + 决策 → 推送老板
```

### 模式C: 竞品应急（Reactive）

```
RADAR 发现竞品大幅降价
    │
    CONDUCTOR 启动应急响应:
    ├── BLADE: 价格弹性分析 + 应对方案
    ├── PULSE: 实时监控自身数据变化
    │         ↓
    ORACLE: 快速决策（跟进/不跟/差异化）→ 推送老板
```

## 五、Blackboard（共享状态板）

所有Agent通过Blackboard实现状态共享，**不直接通信**。

```
blackboard/
├── TASKS.md          # 当前任务状态机
├── DECISIONS.md      # 已确认的决策记录（不可覆盖）
├── MARKET-SIGNALS.md # RADAR 写入的市场信号
├── PRODUCT-DB.md     # 产品数据库（SKU/成本/参数）
├── COMPETITOR-MAP.md # 竞品图谱
├── METRICS.md        # 核心运营指标快照
└── ALERTS.md         # 异常告警队列
```

**写入规则**：
- 每个Agent只写自己的职责域
- 写入必须带时间戳 + 数据源标注
- DECISIONS.md只有CONDUCTOR和ORACLE有写入权
- 冲突检测: 同一字段被两个Agent写入时，CONDUCTOR仲裁

## 七、飞轮架构 vs 线性Pipeline

| 架构维度 | 线性Pipeline | 飞轮架构（电商团队） |
|---------|---------|---------|
| 执行拓扑 | 严格顺序 Phase 1→N | 飞轮 + 事件驱动，支持并行 |
| 迭代模式 | 单次执行 | 持续迭代（PULSE反馈循环） |
| 冲突处理 | 无（线性传递） | CONDUCTOR仲裁协议 |
| 状态管理 | Blackboard单向写入 | Blackboard双向读写 + 冲突检测 |
| 触发机制 | 人工触发 | 人工 + 定时 + 事件驱动三种 |
| 输出形态 | 文档（终态） | 决策 + 行动指令（持续态） |
| 质量门控 | 固定评分标准 | 置信度 + Kill Criteria + A/B验证 |
| 时间维度 | 无（一次性） | 周期性复盘 + 实时监控 |

## 九、扩展点

1. **多平台适配**：每个角色模板支持平台参数（amazon/taobao/shopify），内容策略和指标体系自动切换
2. **Cron集成**：PULSE可配置为定时任务，自动采集数据并触发异常告警
3. **记忆系统**：每次决策循环的结果写入memory，下次迭代时自动加载历史context
4. **多品管理**：Blackboard按SKU分区，支持同时运营多个产品线
5. **老板仪表盘**：ORACLE定期输出一页纸运营摘要，推送Telegram
