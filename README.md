# 互联网大厂架构与 SRE 体系

> 本文档总结互联网大厂的典型技术架构和 SRE（Site Reliability Engineering）体系实践

---

## 🏢 互联网大厂典型架构

### 1. 整体架构演进

```
单机 → 垂直拆分 → 微服务 → 云原生 → Service Mesh
```

**主流架构分层：**

```
┌─────────────────────────────────────┐
│         接入层 (CDN + WAF)           │
├─────────────────────────────────────┤
│         网关层 (API Gateway)         │
├─────────────────────────────────────┤
│         业务中台 (微服务集群)         │
│  ┌─────┬─────┬─────┬─────┬─────┐   │
│  │用户 │订单 │支付 │商品 │库存 │... │
│  └─────┴─────┴─────┴─────┴─────┘   │
├─────────────────────────────────────┤
│         数据中台                      │
│  ┌─────┬─────┬─────┬─────┐         │
│  │ MySQL│ Redis│  MQ │ ES │         │
│  └─────┴─────┴─────┴─────┘         │
├─────────────────────────────────────┤
│         基础设施工具链               │
│  监控/日志/CI-CD/配置中心/服务发现   │
└─────────────────────────────────────┘
```

### 2. 核心组件

| 层级 | 典型技术栈 |
|------|-----------|
| **接入层** | Nginx, LVS, F5, CDN (Cloudflare/阿里云) |
| **网关层** | Kong, APISIX, Spring Cloud Gateway, 自研网关 |
| **服务框架** | Dubbo, Spring Cloud, gRPC, Thrift |
| **服务发现** | Nacos, Consul, Etcd, ZooKeeper |
| **配置中心** | Apollo, Nacos, Consul |
| **消息队列** | Kafka, RocketMQ, Pulsar, RabbitMQ |
| **缓存** | Redis Cluster, Codis, Memcached |
| **数据库** | MySQL (分库分表), PostgreSQL, TiDB |
| **搜索引擎** | Elasticsearch, Solr |
| **监控** | Prometheus + Grafana, Zabbix, 自研监控 |

## 👨‍💻 SRE 体系 (Site Reliability Engineering)

### SRE 核心理念

> **SRE = 用软件工程的方式解决运维问题**

Google 首创，核心目标：**在快速迭代和系统稳定性之间找平衡**

### 1. SRE 关键指标

| 指标 | 说明 | 典型目标 |
|------|------|---------|
| **SLA** (Service Level Agreement) | 服务可用性承诺 | 99.9% ~ 99.99% |
| **SLO** (Service Level Objective) | 内部目标，比 SLA 更严格 | 99.95% ~ 99.999% |
| **SLI** (Service Level Indicator) | 实际测量的指标 | 请求成功率、延迟等 |
| **Error Budget** (错误预算) | 允许的错误时间 = 1 - SLO | 0.1% ~ 0.01% |

**错误预算用法：**
- 预算用完 → 冻结发布，专注稳定性
- 预算有剩余 → 可以加速迭代

### 2. SRE 核心工作

```
┌────────────────────────────────────────────┐
│              SRE 工作分配                    │
├────────────────────────────────────────────┤
│  50% 运维工作 (On-call、故障处理、变更)      │
│  50% 开发工作 (自动化、工具、平台)           │
└────────────────────────────────────────────┘
```

**具体职责：**

| 领域 | 工作内容 |
|------|---------|
| **监控告警** | 指标采集、告警规则、告警收敛、值班响应 |
| **容量规划** | 资源评估、扩缩容策略、成本优化 |
| **变更管理** | 发布流程、灰度策略、回滚机制 |
| **故障管理** | 应急响应、故障复盘、改进跟踪 |
| **自动化** | 自动扩缩容、自愈系统、一键发布 |
| **性能优化** | 延迟优化、吞吐量提升、资源效率 |

### 3. 监控体系 (可观测性)

**三大支柱：**

```
┌─────────────────────────────────────┐
│           可观测性三支柱             │
├─────────────┬─────────────┬─────────┤
│   Metrics   │   Logging   │ Tracing │
│   (指标)    │   (日志)    │ (链路)  │
├─────────────┼─────────────┼─────────┤
│ Prometheus  │   ELK       │ Jaeger  │
│ Grafana     │   Loki      │ Zipkin  │
│ 自研监控    │   Splunk    │ SkyWalking│
└─────────────┴─────────────┴─────────┘
```

**监控分层：**

1. **基础设施监控** - CPU、内存、磁盘、网络
2. **应用监控** - QPS、延迟、错误率、JVM/Go Runtime
3. **业务监控** - 订单量、支付成功率、用户活跃度
4. **用户体验监控** - 首屏时间、API 成功率、崩溃率

### 4. 故障响应机制

**值班制度 (On-call)：**
- 7x24 小时轮班
- P0/P1 故障 5 分钟内响应
- 升级机制：一线 → 二线 → 负责人 → 管理层

**故障分级：**

| 级别 | 影响范围 | 响应时间 | 复盘要求 |
|------|---------|---------|---------|
| **P0** | 核心业务完全不可用 | 5 分钟 | 24 小时内复盘 |
| **P1** | 核心业务部分不可用 | 15 分钟 | 48 小时内复盘 |
| **P2** | 非核心业务受影响 | 1 小时 | 周内复盘 |
| **P3** | 轻微影响 | 4 小时 | 可选复盘 |

**故障复盘 (Post-mortem)：**
- **不追责文化** - 关注系统性问题，不找替罪羊
- **5 Why 分析** - 连续问 5 个为什么找到根因
- **改进项跟踪** - 每个问题必须有 Owner 和 Deadline

### 5. 变更管理

**发布策略：**

```
开发环境 → 测试环境 → 预发环境 → 生产环境
                        ↓
              灰度发布 (1% → 5% → 20% → 100%)
```

**关键实践：**
- **自动化发布** - CI/CD 流水线 (Jenkins, GitLab CI, ArgoCD)
- **蓝绿部署** - 两套环境切换
- **金丝雀发布** - 逐步放量
- **一键回滚** - 5 分钟内完成回滚

### 6. 容量管理

**容量规划流程：**
1. 业务预测 (DAU、订单量增长)
2. 压力测试 (全链路压测)
3. 资源评估 (CPU、内存、带宽)
4. 弹性策略 (自动扩缩容)

**大厂实践：**
- 双 11/618 前全链路压测
- 常态化压测 (每周/每月)
- 混沌工程 (Chaos Engineering) - 主动注入故障验证韧性
